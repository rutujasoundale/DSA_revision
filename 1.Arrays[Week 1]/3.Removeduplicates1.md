
```java
class Solution {
    public int removeDuplicates(int[] nums) {
        if(nums.length==0){
            return 0;
        }
        int k=1;
        for(int i=1;i<nums.length;i++){
            if(nums[i]!=nums[k-1]){
                nums[k]=nums[i];
                k++;
            }
        }
        return k;
    }
}
```
Complexity
Time: O(n)
Space: O(1)
Interview Focus

This problem is a classic slow/fast pointer (read/write pointer) problem.

The interviewer is checking whether you can:

Read each element once.
Keep only unique elements.
Modify the array in-place.
Return the count of unique elements.
Common Follow-up Questions
Why do we initialize k = 1 instead of 0?
Because the first element is always unique in a non-empty sorted array.
Why compare with nums[k-1] instead of nums[i-1]?
nums[k-1] is the last unique element kept. nums[i-1] could be a duplicate that we're skipping.
What if the array is not sorted?
This approach won't work directly because duplicates won't necessarily be adjacent. You'd typically need a HashSet, which uses O(n) extra space.

This problem and Remove Element (LeetCode 27) are excellent practice because they teach the same in-place read/write pointer pattern, which appears frequently in coding interviews.
