It is sorting based approach so its 
Time complexity:O(nlongn)
Space:O(1)
Why n-i?
Because after sorting, there are exactly n - i papers from index i to the end. If the current citation count is at least n - i, then all papers after it also have at least that many citations, satisfying the H-index definition.
### Approach 2:Create extra array,bucket count

```java
class Solution {
    public int hIndex(int[] citations) {
        int n = citations.length;
        int[] count = new int[n + 1];

        // Count frequencies
        for (int c : citations) {
            if (c >= n)
                count[n]++;
            else
                count[c]++;
        }

        int papers = 0;

        // Traverse from highest possible H-index
        for (int i = n; i >= 0; i--) {
            papers += count[i];

            if (papers >= i)
                return i;
        }

        return 0;
    }
}
```
Complexity
Time: O(n)
Space: O(n)
Why does papers += count[i] work?

Suppose:

count

0 :1
1 :1
2 :0
3 :1
4 :0
5 :2

At i = 5:

papers = 2

There are 2 papers with at least 5 citations.

At i = 4:

papers = 2 + count[4]
       = 2

There are still 2 papers with at least 4 citations.

At i = 3:

papers = 2 + count[3]
       = 3

Now there are 3 papers with at least 3 citations, so 3 is a valid H-index.

Interview Tip

If asked why we cap citations at n, explain:

The H-index cannot exceed the total number of papers. Whether a paper has n, 100, or 10,000 citations, it contributes the same toward any H-index calculation greater than or equal to n. So we safely group all citation counts >= n into bucket n.
